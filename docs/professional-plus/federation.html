Multi-Site Federation

A coordinating SysManage server aggregates host inventory, compliance posture, vulnerability exposure, and dispatched commands across many subordinate site servers running independently at each data center, branch office, or cloud region.

Overview

Large enterprises operate data centers, branch offices, and cloud regions across multiple geographic locations. Managing thousands of hosts from a single SysManage server creates scalability bottlenecks, network latency issues, and single-point-of-failure risk.

The federation Pro+ Enterprise SKU introduces a hierarchical architecture: many subordinate site servers operate autonomously at each site, while a Federation Controller aggregates data and dispatches commands across the entire enterprise. The controller never talks to agents directly — everything flows through site servers, preserving local-autonomy semantics if a network partition isolates a site.

Engine Modules

Federation ships as two Pro+ engines in one Enterprise license — a server runs exactly one role at a time, picked by role: in sysmanage.yaml.

Federation Controller Engine

Loaded on the coordinating server. Owns the site registry, accepts upstream rollup pushes from sites, exposes cross-site host search, drives policy push and command dispatch, and writes the federation audit log.

  • Site server registry: enroll, suspend, resume, remove
  • Secure enrollment workflow with mutual-TLS certificate pinning
  • Host-directory tier (~1 KB × 1M-host target ≈ 1 GB)
  • Aggregate rollups: host counts, compliance scores, CVE exposure
  • Polymorphic policy push (update profiles, firewall roles, compliance baselines) with stale-version detection
  • Command dispatch with a strict FSM (queued → in-progress → terminal)

Federation Site Engine

Loaded on every subordinate site server. Handles the inbound side of the federation protocol (enroll, accept policy pushes, accept dispatched commands) and drains an outbound sync queue back upstream.

  • Coordinator enrollment + handshake
  • Upstream data sync (host deltas, compliance + vulnerability snapshots)
  • Downstream policy + command inboxes
  • Offline queue with deduplication-on-replay
  • Local autonomy mode — agents continue reporting and commands run locally when the coordinator is unreachable

Data Architecture

Federation data is split into three tiers, deliberately partitioned so the coordinator scales linearly with sites rather than with hosts:

  1. Aggregate tier (coordinator) — one row per site per metric. Host count, healthy/unhealthy ratio, compliance %, top CVEs by severity, alert counts, last-sync timestamp.
  2. Host-directory tier (coordinator) — one row per host fleet-wide, holding only the columns operators filter on (hostname, IP, OS, status, geo, tags). Sized for ~1 KB × 1M hosts.
  3. Detail tier (sites) — full software inventory, certificate chains, audit log bodies, OS-specific facts. Never replicated upstream; drill-down queries are proxied to the originating site.

Coordinator UI

When the federation controller engine is loaded, several new pages become available in the SysManage web UI:

  • Sites (/sites) — card grid of enrolled sites with status traffic light, last-sync, host count. "Enroll Site" button generates a one-time enrollment token for delivery to the new site server.
  • Site detail (/sites/:siteId) — per-site metadata, connection card, and lifecycle actions (Suspend / Resume / Remove). "See hosts" deep-links into the Hosts page filtered by this site; "View audit log" deep-links the audit log viewer pre-filtered.
  • Sites map (/sites/map) — geographic Leaflet map of every site at its operator-supplied (lat, lng). Click markers for popup details + deep-link into site detail.
  • Policies (/federation/policies) — CRUD on coordinator-defined policies (update profiles, firewall roles, compliance baselines). Assign to multiple sites, push immediately, deactivate.
  • Federation audit log (/audit/federation) — every cross-site operation (enrollment, suspend/resume/remove, policy push, command dispatch) with URL-shareable filters by site, operation, and actor.
  • Sync timeline — the site detail page renders a per-site upstream-sync latency sparkline, the site's reported SysManage version and loaded engine capabilities, and an "Operating independently" banner when the site reports that it is offline from the coordinator (local-autonomy mode).
  • Per-site actions — the site detail header offers a one-click Dispatch command (batch command to the site's hosts) and Push policies (re-queue every policy assigned to the site for re-delivery on the next push tick). Both are gated to actively enrolled sites.
  • Cross-site reports (/reports#federation) — a Federation tab on the Reports page with a site multi-select that produces a cross-site rollup table (per-site host counts, worst compliance baseline, CVE-severity counts) plus enterprise-wide totals.

Connection Health & Local Autonomy

Each site server continuously classifies the health of its own uplink to the coordinator and keeps operating regardless of that uplink's state. The site engine’s sync tick records every attempt and derives a connection state:

  • online — the last upstream sync succeeded.
  • degraded — one or two consecutive sync failures; usually a transient blip, not worth paging on.
  • offline — three or more consecutive failures. The site enters local autonomy mode: agents keep reporting, OS upgrades keep running, and local deltas keep queuing for replay — nothing blocks on the coordinator.

While offline, the site does not hammer a down coordinator every interval — it backs off with an exponential, capped reconnect schedule and resumes draining its queue automatically once the coordinator answers. On reconnect, deduplication-on-replay ensures queued deltas don’t create duplicates upstream.

Every tick the site also reports lightweight metadata upstream — its SysManage version, active-host count and OS breakdown, loaded engine capabilities, and its self-reported connection state — which the coordinator caches on the site’s registry row and surfaces on the site detail page.

Rollup Alerting

The coordinator evaluates every enrolled site against three built-in cross-site conditions on each push-worker cycle, opening, refreshing, or auto-resolving alerts so there is at most one open alert per site per condition:

  • Site offline — a site hasn’t synced within a multiple of its configured sync interval.
  • Compliance below threshold — a compliance baseline’s score dropped under a configured percentage.
  • Vulnerabilities high — a site’s critical-CVE count crossed a configured threshold.

The thresholds for all three conditions are operator-configurable. An Alert thresholds card on the Federation Policies page lets you override any subset; a blank field reverts that threshold to its built-in default. Open alerts appear on the relevant site’s detail page with an acknowledge action.

Federation-Aware Dynamic Secrets

The coordinator owns the master OpenBAO/Vault, so restricted sites never need direct Vault access. A site can request a short-lived dynamic credential on behalf of one of its hosts, and the coordinator issues and rotates it centrally.

  1. The site enqueues a lease request upstream (it never calls the coordinator directly — queue-everything), naming the host, the secret, the Vault backend role, and a TTL.
  2. The coordinator records the request and, on its next reconcile pass, issues the lease from the master Vault.
  3. The lease result is echoed back to the requesting site’s inbox so the site engine can deliver the credential to the host over the agent’s secure channel.

A single coordinator-side reconcile loop services every site’s leases — issuing pending requests, expiring leases whose TTL has elapsed, and pruning old terminal records. There are no per-site sweepers because every lease lives in the one master Vault. Leases are viewable (and revocable) at /api/v1/federation/secret-leases.

The credential value is never persisted in the federation tables — only the Vault lease id (for renew/revoke) and non-sensitive metadata such as a generated username.

Enrolling a Site

  1. On the coordinator's Sites page, click Enroll Site and fill in name, URL, and an optional location label. The default token TTL is 24 hours.
  2. Copy the displayed enrollment token once — it is not retrievable afterwards.
  3. On the new site server, run the federation enroll command with the coordinator URL + the token. The site engine presents the token over mutual TLS to complete the handshake.
  4. After successful handshake, the site flips to enrolled status on the coordinator, the token hash is scrubbed, and the enrolled_at timestamp is stamped.

If the token leaks or expires before the site completes enrollment, use Regenerate token on the site detail page to invalidate the old one and issue a fresh value.

Licensing

Federation is part of the SysManage Enterprise tier. The Enterprise license entitles a deployment to load either the federation controller engine or the federation site engine on any server — the role is configured in sysmanage.yaml. On Community / Professional installs, the federation API endpoints return 403 pro_plus_required and the UI renders an Enterprise upsell on every federation page.

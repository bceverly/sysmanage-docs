Multi-Site Federation

A coordinating SysManage server aggregates host inventory, compliance posture, vulnerability exposure, and dispatched commands across many subordinate site servers running independently at each data center, branch office, or cloud region.

Overview

Large enterprises operate data centers, branch offices, and cloud regions across multiple geographic locations. Managing thousands of hosts from a single SysManage server creates scalability bottlenecks, network latency issues, and single-point-of-failure risk.

The federation Pro+ Enterprise SKU introduces a hierarchical architecture: many subordinate site servers operate autonomously at each site, while a Federation Controller aggregates data and dispatches commands across the entire enterprise. The controller never talks to agents directly — everything flows through site servers, preserving local-autonomy semantics if a network partition isolates a site.

Engine Modules

Federation ships as two Pro+ engines in one Enterprise license — a server runs exactly one role at a time, set via Settings → Server Role (stored in the database, not a config file).

Federation Controller Engine

Loaded on the coordinating server. Owns the site registry, accepts upstream rollup pushes from sites, exposes cross-site host search, drives policy push and command dispatch, and writes the federation audit log.

  • Site server registry: enroll, suspend, resume, remove
  • Secure enrollment workflow with mutual-TLS certificate pinning
  • Host-directory tier (~1 KB × 1M-host target ≈ 1 GB)
  • Aggregate rollups: host counts, compliance scores, CVE exposure
  • Polymorphic policy push (update profiles, firewall roles, compliance baselines) with stale-version detection
  • Command dispatch with a strict FSM (queued → in-progress → terminal)

Federation Site Engine

Loaded on every subordinate site server. Handles the inbound side of the federation protocol (enroll, accept policy pushes, accept dispatched commands) and drains an outbound sync queue back upstream.

  • Coordinator enrollment + handshake
  • Upstream data sync (host deltas, compliance + vulnerability snapshots)
  • Downstream policy + command inboxes
  • Offline queue with deduplication-on-replay
  • Local autonomy mode — agents continue reporting and commands run locally when the coordinator is unreachable

Data Architecture

Federation data is split into three tiers, deliberately partitioned so the coordinator scales linearly with sites rather than with hosts:

  1. Aggregate tier (coordinator) — one row per site per metric. Host count, healthy/unhealthy ratio, compliance %, top CVEs by severity, alert counts, last-sync timestamp.
  2. Host-directory tier (coordinator) — one row per host fleet-wide, holding only the columns operators filter on (hostname, IP, OS, status, geo, tags). Sized for ~1 KB × 1M hosts.
  3. Detail tier (sites) — full software inventory, certificate chains, audit log bodies, OS-specific facts. Never replicated upstream; drill-down queries are proxied to the originating site.

Coordinator UI

When the federation controller engine is loaded, several new pages become available in the SysManage web UI:

  • Sites (/sites) — card grid of enrolled sites with status traffic light, last-sync, host count. "Enroll Site" button generates a one-time enrollment token for delivery to the new site server.
  • Site detail (/sites/:siteId) — per-site metadata, connection card, and lifecycle actions (Suspend / Resume / Remove). "See hosts" deep-links into the Hosts page filtered by this site; "View audit log" deep-links the audit log viewer pre-filtered.
  • Sites map (/sites/map) — geographic Leaflet map of every site at its operator-supplied (lat, lng). Click markers for popup details + deep-link into site detail.
  • Policies (/federation/policies) — CRUD on coordinator-defined policies (update profiles, firewall roles, compliance baselines). Assign to multiple sites, push immediately, deactivate.
  • Federation audit log (/audit/federation) — every cross-site operation (enrollment, suspend/resume/remove, policy push, command dispatch) with URL-shareable filters by site, operation, and actor.
  • Sync timeline — the site detail page renders a per-site upstream-sync latency sparkline, the site's reported SysManage version and loaded engine capabilities, and an "Operating independently" banner when the site reports that it is offline from the coordinator (local-autonomy mode).
  • Per-site actions — the site detail header offers a one-click Dispatch command (batch command to the site's hosts) and Push policies (re-queue every policy assigned to the site for re-delivery on the next push tick). Both are gated to actively enrolled sites.
  • Cross-site reports (/reports#federation) — a Federation tab on the Reports page with a site multi-select that produces a cross-site rollup table (per-site host counts, worst compliance baseline, CVE-severity counts) plus enterprise-wide totals.

Connection Health & Local Autonomy

Each site server continuously classifies the health of its own uplink to the coordinator and keeps operating regardless of that uplink's state. The site engine’s sync tick records every attempt and derives a connection state:

  • online — the last upstream sync succeeded.
  • degraded — one or two consecutive sync failures; usually a transient blip, not worth paging on.
  • offline — three or more consecutive failures. The site enters local autonomy mode: agents keep reporting, OS upgrades keep running, and local deltas keep queuing for replay — nothing blocks on the coordinator.

While offline, the site does not hammer a down coordinator every interval — it backs off with an exponential, capped reconnect schedule and resumes draining its queue automatically once the coordinator answers. On reconnect, deduplication-on-replay ensures queued deltas don’t create duplicates upstream.

Every tick the site also reports lightweight metadata upstream — its SysManage version, active-host count and OS breakdown, loaded engine capabilities, and its self-reported connection state — which the coordinator caches on the site’s registry row and surfaces on the site detail page.

Rollup Alerting

The coordinator evaluates every enrolled site against three built-in cross-site conditions on each push-worker cycle, opening, refreshing, or auto-resolving alerts so there is at most one open alert per site per condition:

  • Site offline — a site hasn’t synced within a multiple of its configured sync interval.
  • Compliance below threshold — a compliance baseline’s score dropped under a configured percentage.
  • Vulnerabilities high — a site’s critical-CVE count crossed a configured threshold.

The thresholds for all three conditions are operator-configurable. An Alert thresholds card on the Federation Policies page lets you override any subset; a blank field reverts that threshold to its built-in default. Open alerts appear on the relevant site’s detail page with an acknowledge action.

Federation-Aware Dynamic Secrets

The coordinator owns the master OpenBAO/Vault, so restricted sites never need direct Vault access. A site can request a short-lived dynamic credential on behalf of one of its hosts, and the coordinator issues and rotates it centrally.

  1. The site enqueues a lease request upstream (it never calls the coordinator directly — queue-everything), naming the host, the secret, the Vault backend role, and a TTL.
  2. The coordinator records the request and, on its next reconcile pass, issues the lease from the master Vault.
  3. The lease result is echoed back to the requesting site’s inbox so the site engine can deliver the credential to the host over the agent’s secure channel.

A single coordinator-side reconcile loop services every site’s leases — issuing pending requests, expiring leases whose TTL has elapsed, and pruning old terminal records. There are no per-site sweepers because every lease lives in the one master Vault. Leases are viewable (and revocable) at /api/v1/federation/secret-leases.

The credential value is never persisted in the federation tables — only the Vault lease id (for renew/revoke) and non-sensitive metadata such as a generated username.

Enrolling a Site

  1. On the coordinator's Sites page, click Enroll Site and fill in name, URL, an optional location label, and the site's identity public key — copied out of band from the site's Settings → Server Role page. The identity key is required: it is what the coordinator uses to authenticate the site at enrollment. The default token TTL is 24 hours.
  2. Copy the displayed enrollment token once — it is not retrievable afterwards.
  3. On the new site server, open Settings → Server Role, set the federation role to Site, and use the Enroll with coordinator action: paste the coordinator URL, the token, and the coordinator's identity public key (copied out of band from the coordinator's Server Role page), then click Enroll. The TLS certificates are still auto-presented and auto-fetched — you never mint or paste a cert — but each side must additionally prove its out-of-band identity key signed the cert being pinned, which is what defeats an enrollment-time man-in-the-middle.
  4. After successful handshake, the site flips to enrolled status on the coordinator, the token hash is scrubbed, and the enrolled_at timestamp is stamped.

If the token leaks or expires before the site completes enrollment, use Regenerate token on the site detail page to invalidate the old one and issue a fresh value.

Deployment & Operations

A federated estate is one coordinator and N site servers. Each server plays exactly one role, chosen at install time and fixed thereafter unless re-provisioned.

Choosing a role

The role is set in the web UI under Settings → Server Role (persisted in the database) — coordinator or site, never both. It is a separate axis from the air-gap server role, so a server can be, e.g., an air-gap collector AND a federation site. A server with the role left at none is a standalone install. The Enterprise license entitles the engines; you pick the role per server.

Bring-up sequence

  1. Stand up the coordinator first and confirm its federation pages load (/sites renders the empty registry rather than the Enterprise upsell).
  2. Stand up each site server as a normal SysManage install with the site engine loaded; let its local agents enroll and report as usual — a site is fully operational before it ever joins the federation.
  3. Enroll each site into the coordinator (see Enrolling a Site above and the mutual-TLS procedure below). Sites can be added incrementally; there is no big-bang cutover.
  4. Define centrally-managed policies (update profiles, firewall roles) on the coordinator and assign them to sites; the coordinator pushes them on its next cycle.

Day-2 operations

  • Watch site health on the Sites page and the per-site sync timeline; an "Operating independently" banner means that site is in local autonomy mode (cut off from the coordinator but running normally).
  • Suspend a site to stop it pulling policy/commands while keeping its registry row; resume to re-attach; remove to disenroll (the row is retained with status=removed for audit).
  • Tune a bandwidth-constrained site's sync_interval_seconds so it syncs less often; the coordinator's offline-alert threshold scales off that interval.
  • Adjust the cross-site alert thresholds (offline multiple, compliance floor, critical-CVE ceiling) on the Federation Policies page; leave a field blank to use the built-in default.

Troubleshooting

  • Site shows offline: check the site engine's last-sync error on its detail page. The site backs off and auto-reconnects on an exponential schedule, so transient outages clear themselves; queued deltas replay (deduplicated) on reconnect.
  • Growing upstream-queue depth: the site is producing deltas faster than it can ship them (usually a coordinator outage). It is not data loss — the queue drains when connectivity returns.
  • Enrollment fails: the most common causes are an expired or already-used token, or an identity-key mismatch (a 401 identity-verification error means the pre-registered identity public key doesn't match the peer's actual key — re-copy it from the peer's Server Role page). Regenerate the token and re-run the enroll action after correcting the key.

Mutual-TLS Enrollment Procedures

Federation traffic is authenticated in both directions. Enrollment establishes the trust material; every subsequent sync and push rides on it.

Certificate pinning

Each server auto-mints a self-signed federation certificate the first time it is given a federation role — the operator never generates or pastes one. At enrollment the site presents its own certificate and fetches the coordinator's from the coordinator's public /api/v1/federation/tls-cert endpoint; the coordinator pins the site's certificate on the registry row and the site pins the coordinator's. Each side thereafter rejects a request whose signature doesn't match the pinned certificate — a leaked bearer token alone can't impersonate either party. The certificate fetched at enrollment is not trusted blindly, though — see Out-of-band identity verification below.

Out-of-band identity verification

Pinning the certificate that is fetched over the network would, on its own, be trust-on-first-use: an attacker who intercepts the enrollment could present its own certificate and be pinned. The enrollment token cannot close this gap because it is a shared secret that travels over the same channel. SysManage therefore anchors enrollment on each server's long-lived Ed25519 identity key, whose private half never leaves the box. Before enrolling, the operator exchanges the two servers' identity public keys out of band (they are public, so any channel works — the Server Role page shows each key and its fingerprint). During enrollment each side signs the exact certificate it is presenting with its identity private key, and the other side verifies that signature against the pre-shared identity public key before pinning. A man-in-the-middle that swaps a certificate cannot forge the matching signature, so enrollment is refused rather than pinned. This verification is strict and mandatory: enrollment fails if an identity key was not pre-registered or the proof does not verify. Because the proof is over the certificate fingerprint, rotating a TLS certificate later is just a re-sign by the same identity key — no second out-of-band exchange.

Bearer tokens

A successful handshake mints two long-lived bearer tokens, one per direction. The site presents its token on every upstream sync POST; the coordinator presents a separate token on every downstream push. Each side stores only the SHA-256 hash of the token it verifies, and the plaintext only of the token it sends — so a database leak on the verifying side never yields a usable credential.

Handshake flow

  1. The operator generates a one-time enrollment token on the coordinator (default 24-hour TTL); only its SHA-256 hash is stored.
  2. On the site, the Enroll with coordinator action fetches the coordinator's certificate and its identity proof, and verifies the proof against the coordinator identity key the operator pre-registered out of band — aborting if it doesn't match. It then POSTs the token to the coordinator with its own certificate and an identity proof attached.
  3. The coordinator validates the token (unused + unexpired), verifies the site's identity proof against the site identity key registered when the site row was created, pins the site certificate, assigns the site a UUID, mints the two bearer tokens, and returns them once. A missing or invalid identity proof is rejected with 401.
  4. The site stores its assigned id + bearers, flips to enrolled, and the coordinator scrubs the enrollment-token hash. From here, only the bearer-authenticated sync/push endpoints are used.

Rotation & revocation

Re-running enrollment (or migrating a site to a different coordinator via clear_enrollment) replaces the bearer tokens and re-pins certificates — the critical step that prevents a stale token from continuing to fire at a former coordinator. Removing a site on the coordinator scrubs its credentials so a disenrolled site can no longer authenticate.

Signature enforcement (HTTP vs HTTPS)

Every sync and push request is signed with the sender's federation private key and verified against the pinned certificate. Enforcement is derived from the deployment posture automatically — no configuration flag. A server running plain HTTP (no certFile/keyFile in sysmanage.yaml) is treated as a development deployment: requests are still signed, but signatures are not rejected, so local testing is never blocked. Configure TLS so the API serves HTTPS and signature verification activates on its own — a request with a missing or invalid signature is then rejected with 401. Production federation deployments should always run HTTPS.

Licensing

Federation is part of the SysManage Enterprise tier. The Enterprise license entitles a deployment to load either the federation controller engine or the federation site engine on any server — the role is chosen in the web UI under Settings → Server Role. On Community / Professional installs, the federation API endpoints return 403 pro_plus_required and the UI renders an Enterprise upsell on every federation page.

GPG Key Management

Store named GPG keys as OpenBAO KV secrets and assign them to a host, or to a specific user on a host. The agent installs or removes the key into the target user's GnuPG keyring over the durable message queue. Private key material never touches the SysManage database or YAML — only the name, fingerprint, and OpenBAO reference are stored.

Overview

GPG Key Management gives you a central catalog of named GPG keys and a controlled way to distribute them across your fleet. Each key is stored as an OpenBAO KV secret so the private material stays inside the vault; SysManage keeps only the key's name, fingerprint, and a reference to its OpenBAO path in the database. From the catalog you assign a key to a host, or to a single user on a host, and the agent installs it into that user's GnuPG keyring for you.

Storage Model

Every GPG key is a named object backed by an OpenBAO KV secret. The split between the vault and the database is deliberate:

  • OpenBAO KV secret holds the private and public key material. It never leaves the vault except when the agent fetches it to install the key.
  • Database holds only the key's display name, its cryptographic fingerprint, and the OpenBAO path reference — never the private material.
  • YAML / config never contains key material either; there is nothing sensitive to leak into /etc/sysmanage.yaml or a backup of it.
  • Fingerprint tracking means the UI and audit trail can identify exactly which key is deployed where without ever reading the secret.

Assignment & Bindings

A key is distributed by creating a binding between the key and a target. Two binding scopes are supported:

Assign to a Host

Bind the key to a host to install it for the default target user on that host. Use this when a key belongs to the machine's automation identity rather than a person.

Assign to a User on a Host

Bind the key to a specific user on a specific host to install it into just that user's GnuPG keyring. The same key can be bound to several host/user pairs; each binding is tracked independently.

Manage bindings from the key's detail view: add a new host or host/user binding, or remove an existing one to schedule the key's removal from that keyring.

Assignment Status

Each binding carries a status that reflects where it is in the install / remove lifecycle:

Pending The binding has been created and an install or remove task is queued for the agent but not yet confirmed.
Installed The agent has imported the key into the target user's GnuPG keyring and confirmed success.
Removed The agent has deleted the key from the target user's keyring; the binding is no longer active.
Failed The install or remove operation did not complete; the binding surfaces the error so it can be retried.

Agent Installation Flow

Distribution runs over the durable message queue, so a target host that is offline simply picks up its task when it reconnects — nothing is lost.

  1. You create or remove a binding in the UI (or via the API).
  2. SysManage enqueues an install or remove task on the target host's durable queue and marks the binding pending.
  3. The agent fetches the key material from OpenBAO, then imports it into — or deletes it from — the target user's GnuPG keyring (~/.gnupg).
  4. The agent reports the result; the binding moves to installed, removed, or failed.

Required Permissions

GPG key operations are gated behind a dedicated role:

  • Manage GPG Keys — required to create, assign, remove, and delete GPG keys and their bindings. Users without this role cannot see or act on the GPG Keys page.

Requirements

  • Professional or Enterprise license with the GPG key management module.
  • An OpenBAO vault installation — the private key material is stored only in OpenBAO KV.
  • OpenBSD support requires OpenBAO running on OpenBSD; without a reachable OpenBAO the key material has nowhere to live and assignment is unavailable.
  • GnuPG (gpg) installed on each target host so the agent can manage the user's keyring.

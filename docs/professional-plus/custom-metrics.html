Custom Metrics & Graphs

Define your own metrics the same way Canonical Landscape does with its Custom Graphs: a small script that emits one numeric value, targeted at hosts by tag, run by the agent on a cadence and sampled over time. Each metric renders as an in-UI time-series graph and can drive alerts.

Overview

Custom Metrics & Graphs brings Canonical Landscape "Custom Graphs" parity to SysManage. The model is deliberately simple: you write a small script that prints exactly one numeric value to standard output. You target which hosts run it using host tags. The agent runs the script on a cadence and reports the value back, and SysManage samples those values over time to build a per-host time-series graph. The same series can be used as an alert condition.

The Model

A custom metric is built from four ideas borrowed directly from Landscape Custom Graphs:

  • One numeric value — the script must print a single number to stdout (an integer or a float). That number is the sample for this run.
  • Targeted by host tag — a metric applies to every host carrying one of its target tags, so a new host that gains the tag automatically starts reporting.
  • Run on a cadence — the agent runs the script on a fixed interval (minimum 60 seconds) rather than continuously.
  • Sampled over time — each reported value is stored as a timestamped sample, and the samples form the time series you graph.

Defining a Metric

Create a metric by giving it a name, the script body, and how it should run:

  • Name — a descriptive label used on the graph and in alert conditions.
  • Script — the body that runs on each host and prints one numeric value.
  • Interpreter — the shell or language the script is run with: sh, bash, or python3.
  • Unit — a label for the value (e.g. %, MB, count) shown on the graph axis.
  • Cadence — how often the agent runs the script; the minimum is 60 seconds.
  • Target tags — the host tags this metric applies to.
# Example: number of established TCP connections
# Interpreter: bash   Unit: count   Cadence: 60s   Target tags: web
ss -tan state established | tail -n +2 | wc -l

Time-Series Graph

Every metric renders as an in-UI time-series graph. Each host that carries a target tag contributes its own series, so you can watch a value trend across the fleet or drill into a single host. The graph reads directly from the stored samples — no external tooling is required for the built-in view.

Custom Metrics as Alert Conditions

A custom metric can be used as an alert condition through the alerting engine. Point an alert at a metric and set a threshold — when a host's latest sample crosses it, the alerting engine fires through its configured channels (email, webhook, Slack, Teams) with the usual cooldown and acknowledge / resolve handling.

Sample Retention

Samples are retained for a configurable window so the built-in graph stays responsive and storage stays bounded. Older samples age out of the retention window; for long-horizon history and heavier analysis, forward the series to Grafana (see below).

Grafana for Heavy Analysis

The in-UI graph is designed for quick, at-a-glance trends. For heavy analysis — cross-metric correlation, long retention, custom dashboards, and shared views — use the Grafana integration provided by the Observability Engine. Grafana reads the same underlying data, so a metric you define here can be charted alongside the rest of your fleet telemetry.

Grafana & Prometheus Setup

To chart custom metrics in Grafana, SysManage exposes them to the existing Prometheus → Grafana pipeline. The steps below wire a Prometheus scrape job to the server, restrict access to that endpoint, and verify the data reaches Grafana.

The Endpoint

The SysManage server publishes the latest value of each custom metric at GET /metrics/custom-metrics in Prometheus text-exposition format. Each metric is emitted as a gauge named sysmanage_custom_metric_value carrying metric, host, and unit labels (plus a tenant label when the server runs in multi-tenant mode). The endpoint is unauthenticated, following the usual Prometheus-scrape convention.

Add a Prometheus Scrape Job

Add a scrape job that points Prometheus at the endpoint, then reload Prometheus so the job takes effect:

scrape_configs:
  - job_name: sysmanage-custom-metrics
    metrics_path: /metrics/custom-metrics
    scheme: https            # http if the server is not behind TLS
    scrape_interval: 60s     # samples update at the agent cadence (min 60s)
    static_configs:
      - targets: ['sysmanage-host:port']

Restrict Access (Important)

Because the endpoint is unauthenticated and shares the same port as the rest of the API, restrict it at the nginx reverse proxy so only the Prometheus host can reach it. Use a path-level allow/deny rule at nginx — do not firewall the port itself, which would block the entire API:

location = /metrics/custom-metrics {
    allow  10.0.0.5;   # Prometheus host IP
    deny   all;
    proxy_pass http://127.0.0.1:8080;   # same upstream as the rest of the API
}

Verify

In Prometheus, open Status → Targets and confirm the sysmanage-custom-metrics job is UP. Query sysmanage_custom_metric_value to see the samples. From there the metric is graphable in Grafana through the existing Prometheus datasource — the one SysManage auto-provisions — so your custom metrics appear alongside the rest of your fleet telemetry.

Requirements

  • Professional or Enterprise license with the custom metrics module.
  • The chosen interpreter (sh, bash, or python3) available on each target host.
  • Host tags configured so metrics can be targeted at the right hosts.
  • Optional: the Observability Engine's Grafana integration for heavy analysis and long-horizon history.

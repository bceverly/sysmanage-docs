Content Lifecycle Management

Satellite-style content views, lifecycle environments, and gated promotion. Turn your repository mirrors into versioned, filtered, promotable content that flows Library → Dev → Test → Prod with instant rollback, air-gap export, federated site sync, and version diffs.

Overview

The content_lifecycle_engine Enterprise module builds on the repository mirroring snapshot substrate. Flat mirrors become content views: named, filtered, versioned selections of repositories. Publishing a content view materializes an immutable version on the mirror host, which you then promote along an ordered environment path. As with the other engines, the OSS server owns the API, the schema, and the license gate (every endpoint returns HTTP 402 without the engine loaded); the engine emits the apply_deployment_plan messages the mirror-host agent runs. No agent code changes are needed.

Content-view definitions (environments, views, filters, versions) are platform truth in the shared partition; the per-tenant promotion state (which version is bound into which environment, the audit trail, and site subscriptions) lives in the tenant partition, cross-referencing the shared IDs.

Lifecycle Environments

An environment is one stage on an ordered promotion path — typically Library → Dev → Test → Prod. The very first environment you create becomes the Library: the path root and publish target. It always sits at position 0 and cannot be deleted. Other environments reorder freely among the remaining positions. Promotion only ever moves content forward along the path.

Content Views & Publishing

A content view is a named selection of repository mirrors (all on one mirror host in this release). Publishing is an asynchronous agent job: for each mirror, the engine takes the latest snapshot, applies the view's filters, regenerates native repository metadata, and freezes the result read-only as version v{n}. A published version is immutable — re-publishing produces v{n+1}, never a mutation. Each view keeps a bounded number of published versions (keep_versions, default 5); pin-aware garbage collection reclaims older, unbound versions while never touching a version an environment is bound to.

Filters

Filters narrow the content captured into a published version. They are applied to the materialized tree before metadata regeneration:

  • Allow list — keep only the named packages.
  • Deny list — remove the named packages.
  • By date — freeze content at a point in time (drop packages newer than a date).
  • Advisory cutoff — include only advisories up to a date (resolved against the advisory catalog).
  • Security only — keep only security-fix packages.

The advisory-coupled filters (advisory cutoff, security only) require the Advisory engine, which resolves them to concrete allow/deny package sets at publish time.

Promotion & Rollback

The promotion lane on each content view shows which version currently occupies each environment. Promoting moves the version bound in one environment into a later one — a pure binding rebind, since a single mirror host serves every environment (no bytes move). Each rebind is captured in an append-only audit trail (publish / promote / rollback).

Rollback is instant and reversible: an environment snaps back to the version it held before its current one. The prior version's bytes are retained (pinned) precisely so rollback never needs to rebuild — and the version you rolled off of becomes the new rollback target, so you can roll forward again.

Serving & Client Repoint

Enabling serving provisions nginx on the mirror host so each environment is reachable at a stable URL — http://<mirror-host>/content-views/{view}/{environment}. That URL is a symlink to the environment's currently-bound version, which promotion atomically re-points, so consuming hosts never change their configuration when content is promoted. Repointing a host rewrites its APT/DNF/zypper/pkg client configuration to install from an environment URL instead of the raw upstream, reusing the same override mechanism as default package mirrors.

Composite Content Views

A composite content view composes several other content views into one published version — for example a base-OS view plus a security view plus an application view. Publishing a composite combines each component's newest published version into a single immutable store, and records exactly which component versions it used in the version's manifest for reproducibility. Composites promote, serve, and roll back exactly like a regular view.

Air-Gap Media Export

Any published version can be exported to signed, immutable air-gap media. The export reuses the air-gap collector to burn the frozen version store into an ISO with an embedded, signed manifest — which the air-gap repository server ingests and serves unchanged. Export runs are tracked with status (QUEUED → BUILDING_ISO → COMPLETE) just like a collection run. This needs the air-gap collector engine in addition to the content-lifecycle engine.

Federated Site Sync

On a federation coordinator, each lifecycle environment can be subscribed to by one or more sites. When a version is promoted into a subscribed environment, the coordinator announces it to each site over the existing (bearer-authenticated) federation channel; the site's mirror host then pulls the version's bytes over HTTP from the coordinator's serve URL and serves them locally at the same environment URL. This is the one place federation carries content bytes rather than metadata only — so a promotion at the coordinator propagates the actual content to every subscribed site.

Version Diff

From a content view's version history you can diff a version against the previous published one. The mirror host lists the package files of both version stores and the server compares them, storing the added / removed package sets on the newer version. It is read-only — the immutable stores are never touched — and answers “what actually changed between v{n-1} and v{n}” at a glance.

API Reference

Every route below returns HTTP 402 without content_lifecycle_engine loaded, under /api/v1/content-lifecycle.

  • GET/POST /environments, PUT/DELETE /environments/{id}, POST /environments/reorder — manage the lifecycle path
  • GET/POST /content-views, GET/PUT/DELETE /content-views/{id} — manage content views (mirrors, filters, composites)
  • POST /content-views/{id}/publish, GET /content-views/{id}/versions — publish an immutable version; list version history
  • GET /content-views/{id}/bindings, POST .../promote, POST .../rollback, GET .../audit — promotion lane, promote, rollback, audit trail
  • POST /content-views/{id}/serve, POST .../repoint — provision nginx serving; repoint a host's package manager at an environment
  • POST /content-views/{id}/versions/{v}/export, GET /content-views/{id}/exports — build signed air-gap media; list export runs
  • POST/GET /content-views/{id}/versions/{v}/diff — compute and read a version diff
  • GET/POST/DELETE /environments/{id}/subscriptions — manage which federation sites receive an environment's content (coordinator)

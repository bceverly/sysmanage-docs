OS Lifecycle Management

Know exactly which hosts are running an operating system that is approaching — or already past — end-of-life, and drive the distro release-upgrade to fix it. The lifecycle engine tracks per-release EOL dates and orchestrates scheduled, maintenance-window-aware upgrades using each platform's native mechanism.

Overview

Every OS release has a support lifecycle — a date after which the vendor stops shipping security fixes. Running past that date is a compliance and security liability, but it is easy to lose track of across a mixed fleet. OS Lifecycle Management adds an EOL registry and, per host, computes whether its OS release is supported, approaching EOL (within 180 days), end-of-life, or unknown.

Lifecycle data is sourced from endoflife.date and covers Ubuntu, Debian, RHEL and its rebuilds (AlmaLinux, Rocky), Fedora, SUSE / openSUSE, FreeBSD, and more.

The lifecycle registry is shared

An OS release's EOL date is identical for every customer — Ubuntu 22.04's end-of-life is the same everywhere — so the lifecycle registry is stored once and shared across all tenants, exactly like the CVE and advisory catalogs. Per-host EOL status is computed by joining that registry against each tenant's host inventory; no per-host EOL rows are stored. Refreshing the registry pulls the latest data server-wide, never per tenant.

Fleet dashboard

The OS Lifecycle page in the main navigation summarises the fleet: how many hosts are supported, approaching EOL, end-of-life, or of unknown status, followed by a table of the hosts at risk — those approaching or past EOL — with the days remaining (or overdue) and the recommended upgrade target. Refresh EOL data re-pulls the registry from endoflife.date.

Per-host EOL & release-upgrade

Open a host's detail page and select the OS Lifecycle tab to see its support status, EOL date, days remaining, and — when the host is approaching or past EOL — a one-click Upgrade to the recommended next release.

The engine infers the correct upgrade method from the host's OS and runs a set of pre-flight checks (recorded on the job) before anything is dispatched:

  • Ubuntu / Debian — do-release-upgrade
  • RHEL / Fedora / AlmaLinux / Rocky — dnf system-upgrade
  • SUSE / openSUSE — zypper dup
  • FreeBSD — freebsd-update

Scheduled & maintenance-window aware

A release-upgrade is dispatched through SysManage's normal store-and-forward command queue, so it is audited and honours any maintenance window configured for the host — the upgrade is held until the host's next allowed window. You can also give a job an explicit scheduled time, and the queue defers delivery until then.

Relationship to patching

Release-upgrades move a host to a whole new OS release; day-to-day patching of the current release is covered by Advisory Management and Vulnerability Scanning. Use the lifecycle dashboard to catch the hosts those views can't help — the ones whose OS itself has gone out of support.

Edition Requirements

OS Lifecycle Management is a Professional+ capability provided by the lifecycle_engine module. In the Community Edition the lifecycle API returns HTTP 402 / 403 and the OS Lifecycle page and host tab are hidden. Upgrade to a Professional or higher tier to enable it.

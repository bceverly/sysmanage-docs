Logging Configuration

Send SysManage server and agent logs to the operating system's native logging service, and manage logging centrally from the database — with changes pushed to agents automatically.

Overview

Both the SysManage server and agent write to a rotating file log by default. In addition, they can emit to the host's platform-native logging system, and the whole fleet's logging can be configured centrally from the Settings → Logging page rather than by editing yaml files on each machine.

Platform-native logging

When enabled, logs are also delivered to the OS-native sink so they appear in the tools your operators already use:

  • Linux: the systemd journal (journalctl), falling back to syslog if the journal library is unavailable.
  • macOS and the BSDs: syslog.
  • Windows: the Windows Event Log.

The target is chosen automatically per platform, or you can pin it explicitly. If the requested sink isn't available (for example a missing optional dependency), SysManage logs a notice and continues with file logging — it never fails to start over logging.

Database-stored logging settings

The Settings → Logging page stores logging configuration in the database, which takes precedence over the yaml files. It has two parts:

  • Server logging — a single card for the server. It only offers native targets valid for the operating system the server is currently running on (for example, it won't offer journald on a Windows server).
  • Agent defaults by operating system — a card per OS family (Linux, Windows, macOS, BSD). Each card sets the default logging for agents of that OS, again offering only the native targets that make sense for it.

There is one card per agent operating system, titled by OS. Each card has a single switch that turns native logging on or off for that OS's agents, plus fields for the native target, the identifier/tag, the log level, and the verbosity. Turning the switch off is saved and pushed to those agents, which then stop writing to the native sink and continue with their rotating file log; turning it on delivers logs to the OS-native sink as well.

Remote syslog forwarding ⭐ PRO+

The local sinks described above — the rotating file log, journald, the local syslog daemon, and the Windows Event Log — are part of the free, open-source edition and stay free. In addition, SysManage can forward its own server and agent logs to a central syslog server so they land alongside the rest of your infrastructure's logs.

Choosing the remote syslog target adds fields for the destination host, port, facility, and protocol (UDP or TCP). This works for both the server's own logs and each OS family's agent defaults, and — like every other logging change — it is pushed to connected agents live and re-applied on reconnect.

Edition: remote syslog forwarding is a Professional capability gated behind the LOG_ROUTING feature. Saving a remote destination without a Professional (or higher) license is rejected with HTTP 402; the local sinks remain fully available in every edition.

How changes reach agents

When you save the Logging page, the server applies its own logging change immediately and pushes the resolved configuration to agents:

  • Connected agents receive the update over the durable message queue and re-apply their logging live — no restart required.
  • Offline agents receive it the next time they reconnect, and every agent is also handed the current configuration when it connects.

Within a short settle time the whole fleet is using the new logging settings. Because the database wins over yaml, a reconnecting agent always converges on the server's configuration.

Precedence

Logging is resolved with the database taking priority: a stored setting always wins; where none exists, the value from the yaml file (or a sane default) is used. This lets you keep a baseline in yaml for a fresh install while managing day-to-day changes centrally from the UI.

Access

The Logging settings page is available to administrators. See the Settings Guide for the surrounding settings pages.

PostgreSQL High Availability

Run the SysManage server against a highly-available PostgreSQL cluster and survive a primary failover without manual intervention.

Overview

SysManage stores all of its state in PostgreSQL. In a high-availability deployment, PostgreSQL runs as a cluster (a primary plus one or more streaming-replication standbys) managed by a cluster manager, and the SysManage server reaches it through a stable endpoint that always resolves to the current primary.

Design principle: survive a failover, do not orchestrate one

The SysManage server is deliberately scoped to survive a failover, not to drive one. Leader election, promotion of a standby, and split-brain protection stay in the cluster manager and its connection router. The server never tries to discover the primary itself — re-implementing primary discovery inside an application is a well-known anti-pattern that fights the cluster manager and causes split-brain writes.

The server does NOT: elect a leader, promote a standby, fence a failed node, or decide which node is writable. Those are the cluster manager's job.

The server DOES: detect that a pooled connection died, discard and reconnect it, briefly retry idempotent work during the promotion gap, and report its database connectivity so an upstream load balancer can route around it.

Reference topology

A supported HA topology has three layers. SysManage only participates in the third.

  • Cluster manager — Patroni, repmgr, pg_auto_failover, or Stolon. Owns leader election, promotion, and fencing.
  • Connection router — HAProxy, PgBouncer, a virtual IP (VIP), or a libpq multi-host connection string. Presents a stable endpoint that always points at the current read-write primary.
  • SysManage server — connects to that stable endpoint and does nothing cluster-aware beyond the pool/retry/health behavior described below.

This mirrors the guidance the federation coordinator already gives (“HA via standard replication plus a load balancer”); the single-server case now matches it.

Connection endpoint strategy

Point the SysManage server's database configuration at a stable endpoint, never at one node's fixed address. There are two supported options; pick one.

Option A (recommended): libpq multi-host DSN

psycopg (psycopg3, the driver SysManage uses) honors libpq's multi-host connection strings. List every cluster node and let libpq walk to the writable primary — no proxy required, and it re-resolves the primary transparently after a failover with zero code changes.

In /etc/sysmanage.yaml, set the database host to the comma-separated node list and request the read-write endpoint:

database:
  user: sysmanage
  password: your-password
  # Comma-separated cluster nodes. libpq tries each and keeps the
  # one that accepts read-write connections (the current primary).
  host: "10.80.0.20,10.80.0.21"
  port: 5432
  name: sysmanage
  # Appended to the connection so libpq only accepts a writable node.
  options: "target_session_attrs=read-write"

After a failover libpq skips the demoted (now read-only or unreachable) node and connects to the newly-promoted primary on the next connection attempt.

Option B: proxy or virtual IP

Alternatively, point the server at a single stable address fronted by HAProxy/PgBouncer or a VIP that the cluster manager moves to the current primary:

database:
  host: "pg-primary.internal"   # VIP / proxy that follows the primary
  port: 5432

Both options keep the primary's identity out of SysManage's configuration — the requirement that makes zero-touch failover possible.

What the server does automatically

These are applied to every database engine the server builds — the bootstrap engine, the registry/shared partitions in collapsed mode, and the per-tenant engines in a multi-tenant deployment.

  • Connection pre-ping (pool_pre_ping) — after a failover the pooled connections are dead sockets. Pre-ping issues a cheap liveness check when a connection is checked out of the pool and transparently discards + reconnects a dead one to the new primary, instead of handing a stale socket to the next request.
  • Connection recycling (pool_recycle, 30 minutes) — caps connection age below typical proxy/server idle timeouts so long-lived connections do not accumulate.
  • Bounded retry during the promotion gap — for the few seconds while a standby is being promoted there is no writable primary, so even a freshly-opened connection can fail. Idempotent units of work retry with bounded exponential backoff until the new primary accepts connections. Only idempotent/uncommitted work is retried — a partially-applied transaction is never blindly replayed.
  • Health reporting — GET /api/health/db (readiness) probes database connectivity and returns HTTP 503 when the database is unreachable (200 otherwise), so an external load balancer or orchestrator can route around a server that has lost its database. Point your DB-aware readiness check at this endpoint. GET /api/health (liveness) is intentionally dependency-free — it stays cheap for high-frequency agent/load-balancer polling and does not touch the database.

OpenBAO dynamic database credentials

This section applies only to multi-tenant (Professional+) deployments. A single-tenant server reads a static database password from sysmanage.yaml and is fully covered by the pool and DSN behavior above — it does not lease credentials.

In a multi-tenant deployment, per-tenant database credentials are leased from OpenBAO's database secrets engine: OpenBAO connects to PostgreSQL as an administrator, creates a short-lived role with a TTL, hands it to the server, and revokes it when the lease expires. That means there are two connections to PostgreSQL, and both must survive a failover:

  • Server → PostgreSQL (using the leased credential) — covered by pre-ping, recycle, retry, and the multi-host DSN described above.
  • OpenBAO → PostgreSQL — the database secrets engine's own admin connection, used to create and revoke roles when issuing leases. This is configured independently of the server.

Point OpenBAO's connection at the cluster endpoint

The database secrets engine is configured with its own connection_url. If that URL names a single node and that node is demoted or killed, OpenBAO can no longer mint credentials — every new tenant lease fails even though the PostgreSQL cluster is healthy. Point it at the same stable endpoint the server uses (proxy/VIP, or a libpq multi-host list):

bao write database/config/sysmanage \
  plugin_name=postgresql-database-plugin \
  allowed_roles="tenant-*" \
  connection_url="postgresql://{{username}}:{{password}}@10.80.0.20,10.80.0.21:5432/postgres?target_session_attrs=read-write" \
  username="openbao_admin" password="..."

Roles must exist on the promoted node

A role OpenBAO creates on the primary is a cluster-global object, so physical streaming replication copies it to the standby — a credential minted before a failover keeps working on the promoted node. Verify on a live cluster that a lease requested after a failover also succeeds (OpenBAO reconnected to the new primary and created the role there).

Server-side lease-acquisition retry

Because OpenBAO surfaces a mint failure as an HTTP error (not a driver-level connection error), the server's transient-DB retry does not automatically cover the lease-acquisition call. Requesting a tenant credential during the promotion gap can therefore fail even though the cluster recovers moments later. Treat lease acquisition as its own retry boundary: a bounded, transient-only retry around the OpenBAO lease call closes this gap without masking permanent errors (permission denied, engine not mounted).

Testing a failover locally

The repository ships a script that stands up a complete local HA test environment with libvirt/KVM: a SysManage server VM, an agent VM, and a two-node PostgreSQL 18 cluster, all on an isolated network your host can browse directly.

# Stand up the environment (idempotent)
scripts/buildPostgresHATestNetwork.sh start

# Show VM + network state
scripts/buildPostgresHATestNetwork.sh status

# Tear it all down
scripts/buildPostgresHATestNetwork.sh stop

The build is fully automated end-to-end — every node configures itself through cloud-init the moment it boots, with no manual per-machine steps to copy-paste and get wrong. Each VM's role is derived from its hostname and a self-contained, idempotent setup.sh is injected and run at first boot, so start alone produces a working, failover-ready cluster:

  • pg-primary — installs PostgreSQL 18, opens listen_addresses, sets wal_level=replica and the replication slots/senders, adds the scram-sha-256 pg_hba rules for the app and the replicator, and creates the roles, database, and replication credential.
  • pg-standby — applies the same listen/pg_hba config (a base backup does not copy /etc, so the standby must stand on its own after promotion), takes a pg_basebackup -R -X stream from the primary with .pgpass auth, starts streaming, and then verifies replication is actually flowing (pg_stat_wal_receiver.status = streaming) before declaring itself ready.
  • server — installs SysManage from the Launchpad PPA, waits for the read-write database endpoint, generates its JWT/salt, writes /etc/sysmanage.yaml with the multi-host DSN + target_session_attrs=read-write, runs alembic upgrade head, and starts on http://<server>:8080.
  • agent — installs the agent and points it at the server, so you have live traffic to watch during the failover.

start prints the URL to open in your host's browser and the exact per-node commands (using the real PostgreSQL major version it installed) if you want to inspect or drive replication by hand. To exercise the failover path, take the primary down mid-load — stop PostgreSQL on the primary node (sudo pg_ctlcluster 18 main stop) or power the VM off (virsh destroy pg-primary) — then promote the standby with sudo pg_ctlcluster 18 main promote on it. Confirm the server keeps serving: pooled connections pre-ping, discard, and reconnect to the promoted node, and the rows written before the failover are all present. This is the exact path verified on a live cluster for Phase 15.

Maintenance Windows

Maintenance Windows let you control when SysManage may make changes to your hosts. Update installs and remote commands are held in the server's outbound queue and released only while an allowed window is open, so patching and command execution happen during times you choose rather than the moment someone clicks a button.

Overview

The Maintenance Windows page is reached from the top navigation. It shows every window you have defined and provides a Schedule Maintenance button to create new ones. Each window is one of two kinds:

  • Allow A window during which changes may run. When a host has one or more allow windows, changes are held until one of them is open.
  • Blackout A window during which changes must not run. A blackout always wins: while a blackout is open, changes are held even if an allow window is also open.

Windows are opt-in per host. A host that no window applies to is unrestricted — its changes run immediately, exactly as they did before you defined any windows. This means enabling the feature never silently freezes your fleet; a host is gated only once you scope a window to it.

How gating works

When the server has an update install or remote command queued for a host, it evaluates that host's windows at release time:

  • If an emergency override is active for the host, the change is released immediately.
  • Otherwise, if a blackout window is currently open for the host, the change is held.
  • Otherwise, if the host has allow windows, the change is released only while one of them is open; at other times it is held.
  • Otherwise (no windows apply), the change is released immediately.

Held changes stay queued and are released automatically when the next window opens — nothing is lost or cancelled. Only change actions (update installs and remote commands) are gated; routine control-plane traffic such as configuration pushes is never delayed.

Creating a window

Click Schedule Maintenance to open the editor.

The editor fields are:

  • Name and Description Free text to identify the window.
  • Kind Allow or Blackout, as described above.
  • Recurrence One of:
    • One-time A single window with an absolute UTC start and end.
    • Daily Repeats every day at a local start time for a set duration.
    • Weekly Repeats on the selected days of the week.
  • Timezone The IANA timezone the recurring start time is interpreted in (for example America/New_York). Recurrence is evaluated in this zone, so windows follow local time including daylight-saving changes.
  • Start time and Duration (minutes) For daily and weekly windows. A window may cross midnight.
  • Days of week For weekly windows.
  • Applies to (scope) All hosts, one or more Specific hosts, or Hosts by tag (every host carrying the selected tags). A window may combine several hosts or tags.
  • Enabled A disabled window is retained but ignored during evaluation.

The host maintenance card

Each host's detail page shows a Maintenance Window card that reports the host's current state — Unrestricted, In window, Blocked, or Override active — along with the current or next window, or the active blackout, as applicable. When a host is blocked or in a window, the card offers an Emergency override action.

Emergency override

An emergency override temporarily allows changes to reach a single host regardless of its windows — for shipping an urgent security fix during a blackout, for example. You provide a reason and a duration; the override expires automatically when the duration elapses. Every override is recorded in the audit log with the reason, the acting user, and the expiry, so the decision to bypass change control is always traceable. See the Audit Log Viewer to review overrides.

Edition Requirements

Maintenance Windows — including allow and blackout windows, per-host/tag scoping, and audited emergency overrides — are available in all editions of SysManage, including the Community Edition.
